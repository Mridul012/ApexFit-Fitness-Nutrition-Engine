# ApexFit: Class Diagram

### 1. Overview

The ApexFit Class Diagram shows how the system is structured using different classes. It separates the data (models) from the logic (services), making the system easier to manage and understand. The recommendation system uses simple rules to process user data and give useful suggestions.

---

### 2. Class Diagram

```mermaid
classDiagram
    %% Domain Models (Entities)
    class User {
        +UUID id
        +String email
        +String passwordHash
        +String fullName
        +updateProfile() boolean
    }

    class Goal {
        +UUID id
        +UUID userId
        +String type
        +double targetWeightKg
        +double targetCalories
        +updateTarget(double newCalories)
    }

    class Meal {
        +UUID id
        +UUID userId
        +String name
        +double calories
        +double protein
        +double carbs
        +double fats
        +calculateTotalMacros() double
    }

    class Workout {
        +UUID id
        +UUID userId
        +String type
        +int durationMins
        +double caloriesBurned
        +calculateIntensity() int
    }

    class Progress {
        +UUID id
        +UUID userId
        +double weightKg
        +double bodyFatPercentage
        +Date recordedAt
        +comparePrevious() double
    }

    class Recommendation {
        +UUID id
        +UUID userId
        +String type
        +String message
        +Date generatedAt
        +markAsRead()
    }

    %% Service Layer (Business Logic)
    class AuthService {
        +register(UserDTO dto) User
        +login(Credentials creds) Token
    }

    class MealService {
        +logMeal(MealDTO dto, UUID userId) Meal
        +calculateDailyIntake(UUID userId) SummaryDTO
    }

    class WorkoutService {
        +logWorkout(WorkoutDTO dto, UUID userId) Workout
        +calculateWeeklyExpenditure(UUID userId) double
    }

    class ProgressService {
        +trackProgress(ProgressDTO dto, UUID userId) Progress
        +analyzeTrend(UUID userId) TrendDTO
    }

    class RecommendationService {
        +generateRecommendations(UUID userId) List~Recommendation~
        -evaluateLogs(UUID userId) boolean
        -applyRules(List~Meal~, List~Workout~) Recommendation
    }

    %% Relationships (Models)
    User "1" --> "1" Goal : sets
    User "1" --> "*" Meal : logs
    User "1" --> "*" Workout : tracks
    User "1" --> "*" Progress : records
    User "1" --> "*" Recommendation : receives

    %% Service to Model Dependencies
    AuthService ..> User : handles
    MealService ..> Meal : manages
    WorkoutService ..> Workout : manages
    ProgressService ..> Progress : manages
    RecommendationService ..> Recommendation : creates
    RecommendationService ..> Meal : uses
    RecommendationService ..> Workout : uses
```

---

### 3. Class Descriptions

#### Domain Models (Entities)

* **User**: Stores user details like email, password, and profile information.
* **Goal**: Stores the user’s fitness goal and target values.
* **Meal**: Stores daily food data like calories and nutrients.
* **Workout**: Stores exercise details and calories burned.
* **Progress**: Stores weight and body measurements over time.
* **Recommendation**: Stores suggestions generated by the system.

---

#### Service Layer (Business Logic)

* **AuthService**: Handles user login and registration.
* **MealService**: Handles meal logging and calculates daily intake.
* **WorkoutService**: Handles workout logging and calculates energy usage.
* **ProgressService**: Tracks progress and analyzes changes over time.
* **RecommendationService**: Uses user data and simple rules to generate suggestions.
